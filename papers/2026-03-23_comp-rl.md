## 1) 一句话 Why-read（必填）
- **Key claim/contribution + key observation：** 受神经科学 Complementary Learning Systems 启发，提出将 policy actor 和 experience extractor 在 RL 训练环内**共同演化**（而非静态/外挂记忆），让 extractor 的蒸馏策略随 actor 能力增长同步进化；关键发现：静态经验随训练推进会因分布错位而**变为负资产**，共演化闭环在 4 个 agentic 环境上一致提升 ~10%，且多任务场景增益随任务数扩展（3-task +6.6%, 6-task +8.1%）。

## 2) CRGP 拆解 Introduction（必填）
### C — Context
- RL 用于 LLM agent 时，outcome-based sparse reward 导致样本效率低下：rollout 中的过程性信息（成功策略、失败模式、关键决策点）未被利用。
- 一系列工作开始探索从历史轨迹中蒸馏结构化经验（experience），以 in-context 方式注入 actor 来加速学习。

### R — Related work
- **经验质量线：** ReasoningBank、MemoRAG（维护自优化经验库）；AgentEvolver、SkillRL、Memory-R1（专用 extractor 蒸馏经验）。
- **经验检索线：** MemRL、Memento（多阶段检索启发式）。
- **经验利用线：** MEMOS、MemInsight（直接上下文注入）；AgentEvolver（rollout/optimization 解耦）；Self-Distillation RL（蒸馏到无经验条件下的参数）。

### G — Gap
1. 现有方法将经验视为**静态资源**：经验库固定构建，或 extractor 不随 actor 能力变化做参数更新，导致 actor 越强，经验越过时（distributional misalignment）。
2. 需同时满足三个设计需求未被联合解决：❶ Actor-Extractor 共演化；❷ 经验自动合并/去冲突/去冗余；❸ 异步训练不阻塞 actor rollout。

### P — Proposal
- **Complementary RL**：双模型（actor πθ + extractor πφ）在 RL 训练环内共同演化。
  - Actor 用 outcome reward 优化（split-group GRPO：经验引导 vs 无经验子组分别估优势）。
  - Extractor 用 binary utility reward r(m)∈{-1,+1} 优化（CISPO 目标），信号来自"该经验是否帮助 actor 成功"。
- **ExperienceManager ℋ**：集中式异步管理器，Producer-Consumer 蒸馏 + Periodic Merge + Query Batching + search_and_ask 工具。

## 4) Experiments（必须含具体数字）
### 4.1 Experimental setup
- **任务/数据：**
  - 单任务：MiniHack Room、WebShop、ALFWorld、SWE-Bench（4 个 open-ended agentic 环境）
  - 多任务：MiniHack Room + ALFWorld + WebShop（3-task）；再加 3 个更难任务（6-task）
- **模型/agent 配置：**
  - Actor πθ：Qwen2.5-7B-Instruct（默认）；SWE-Bench 单任务实验用 Qwen3-4B-Instruct-2507
  - Experience Extractor πφ：Qwen3-4B-Thinking-2507（默认）；消融中对比 Qwen3-30B-A3B-Instruct-2507
  - Embedding model fψ 用于经验库语义检索
- **对比基线：**
  - Baseline：无经验的 outcome-based RL
  - Offline Exp.：预构建静态经验库
  - Static Online Exp.：πφ 在线维护经验但不做参数更新
  - Exp. Only：只优化 πφ，固定 πθ
- **评测指标：** Success rate / Reward（held-out eval set）；actions per task（效率指标）

### 4.2 Main result table（必填）

#### 单任务训练（Complementary RL vs Baseline）
| Environment | Performance Gain | Action Efficiency |
|---|---:|---|
| MiniHack Room | **1.3× performance margin** | 1.5× fewer actions |
| ALFWorld | **1.3× performance margin** | 2× fewer actions |
| WebShop | consistent improvement | — |
| SWE-Bench | **+3.0%** | more actions (deeper completion) |

#### 多任务训练（Table 1，3-task mixture）
| Method | w/ exp | w/o exp |
|---|---:|---:|
| Baseline | — | reference |
| Static Online Exp. | < Baseline (gap >10%) | < Baseline |
| Exp. Only | marginal gain | — |

|---|---:|

### 4.3 Analysis experiments（强制"现象+解释"）

- **现象1：** 静态经验（Offline Exp.）初期提供 performance boost，但随训练推进增益消退甚至变为负资产（Figure 3a）。
  **解释（作者）：** Distributional misalignment——静态 ℳ 无法跟踪 πθ 演化的状态-动作分布，导致检索出的经验与 actor 当前能力不匹配（stale guidance）。
  **【标注】（我的判断）：** 这是 agent memory 系统的核心难题：**记忆必须与策略共演化，否则记忆越多反而越有害**。直接挑战 Evo-Memory / ARISE 等只在推理时更新记忆的设计。

- **现象2：** 所有 rollout 都用经验引导时，actor 过早收敛并落后于混合设置（Figure 3b）；跨子组计算 advantage 导致训练崩溃（Figure 3c）。
  **解释（作者）：** Actor 产生对外部经验的过度依赖（over-reliance），无法将经验内化为自身能力。两子组奖励分布不同导致 advantage 估计偏差。
  **【标注】（我的判断）：** "经验引导 + 无经验"子组设计 = **强制 actor 同时学习"用经验"和"不用经验"两种模式**，确保经验被内化而非外挂。这是 Complementary RL 最精巧的设计之一。

- **现象3：** 更大的 extractor（Qwen3-30B-A3B vs 4B-Thinking）在多任务设置中带来一致 +5% 提升（Figure 9a）。
  **解释（作者）：** 更强的 extractor 能提取更具泛化性和信息量的经验，进一步驱动 actor 学习。
  **【标注】（我的判断）：** 暗示 extractor capacity 是系统瓶颈——在实际部署中，值得为经验蒸馏投入更强模型。

- **现象4：** 自蒸馏（self-distillation）初期改善 Complementary RL，但后期训练崩溃（Figure 9b）。
  **解释（作者）：** 可能源于超参选择不当，或应周期性而非每步应用。
  **【标注】（我的判断）：** 自蒸馏 + 共演化 RL 的组合值得继续探索，但需要更细粒度的调度策略。

- **现象5：** Complementary RL 在 MiniHack Room 上减少 1.5× 动作数、ALFWorld 上减少 2× 动作数（Figure 7）。
  **解释（作者）：** 蒸馏后的经验引导 actor 做出更高效的决策，减少无效探索。
  **【标注】（我的判断）：** 动作效率提升 = 经验质量的直接证据。在长 horizon agent 任务中，这比 success rate 更重要。

- **现象6：** 异步框架引入的 rollout 延迟 ≈0（Figure 9c），search time 最高 ~1 秒。
  **解释（作者）：** Query batching + embedding cache + 并行 search workers 消除了经验检索的阻塞延迟。
  **【标注】（我的判断）：** 工程设计值得学习：经验系统的实用性取决于是否能"不减速训练"。

## 5) Why it matters for our work
- **Agentic memory（核心关联）：** Complementary RL 首次证明"经验必须与策略共演化"——静态记忆不仅无用，还有害。这直接挑战了 ReMemR1/MemOCR 中"记忆构建与策略训练分离"的假设，提出了更高标准：记忆系统必须接受端到端梯度反馈。
- **Long-context reasoning：** Split-group GRPO 将"有经验"和"无经验"rollout 分组估优势，确保 actor 既能利用上下文经验又不丧失独立推理能力——这是 long-context agent 设计的关键平衡。
- **Multi-modal agents + RL：** 异步双模型架构（actor + extractor）天然适配多模态场景：视觉/语言 extractor 可以各自演化，通过经验库协调。
- **Retrieval-augmented reasoning：** search_and_ask 机制让 actor 在交互过程中主动查询经验库，类似 RAG 的在线检索，但检索对象是训练过程中持续演化的结构化经验而非静态文档。

### Case 对齐（>=2）
- **Case A（对齐 `2026-03-19_arise.md`）：** 两者都强调"经验/技能必须在 RL 训练环内演化"，但 ARISE 是单模型共享参数（Manager-Worker），Complementary RL 是双模型各自优化。互补视角：ARISE 证明"内化选择"，Complementary RL 证明"外化蒸馏也需共演化"。
- **Case B（对齐 `2026-03-23_trt.md`）：** TRT 在 test-time 通过 Generate-Select-Reflect 迭代累积知识（不更新权重），Complementary RL 在 train-time 通过 actor-extractor 共演化累积经验（更新双模型权重）。前者是推理时自改进，后者是训练时经验驱动学习——两条路可以组合。

## 6) Actionable next step
- [ ] 在 ReMemR1 / MemOCR 框架中引入 experience extractor 的 RL 训练信号：给记忆写入操作分配 utility reward（该记忆是否帮助后续任务成功），验证共演化 vs 静态记忆的差距。
- [ ] 复现 split-group GRPO：在 agent 训练中将 rollout 分为"有记忆"和"无记忆"两组，分别估 advantage，防止 actor 对记忆过度依赖。
- [ ] 将 search_and_ask 机制迁移到多模态 agent：让 actor 在交互过程中主动查询经验库获取 situation-specific 指导，而非仅在 episode 开始时检索一次。
- [ ] 做 extractor capacity scaling 实验：对比 4B / 7B / 30B extractor 对 agent 最终性能的影响，确定最优性价比。
