## 1) 一句话 Why-read（必填）
- **Key claim/contribution + key observation：**XSkill 把可复用知识拆成「task-level skills + action-level experiences」并做视觉对齐的抽取/检索/改写，在 5 个 benchmark、4 个 backbone 上稳定优于 tool-only 与已有 memory 基线；核心观察是 **写入质量（Phase I）比检索花活（Phase II）更关键**。

## 2) CRGP 拆解 Introduction（必填）
### C — Context
- 多模态 agent 虽有工具能力，但仍存在两类瓶颈：
  1) tool use 低效（简单题步骤冗余、难题探索不够深）；
  2) tool orchestration 僵化（单路径执行、跨任务泛化弱）。
- 现实需求：在**不更新参数**前提下，通过历史轨迹持续提升。

### R — Related work
- 经验流（experience memory）：偏局部战术，帮助工具选择与错误恢复。  
- 技能流（skill/workflow memory）：偏结构化流程，帮助规划与编排。  
- 既有工作多依赖文本日志，视觉 grounding 不足；且常局限于 GUI/空间等窄场景。

### G — Research gap
- 缺统一框架同时建模 experience + skill 双流。  
- 多模态场景缺少视觉对齐的知识抽取与检索。  
- 缺 accumulation→inference→feedback 的闭环持续学习机制。

### P — Proposal
- 提出 **XSkill** 双流框架：
  - **Skill Library（Markdown）**：任务级流程与模板；
  - **Experience Bank（JSON）**：条件-动作级战术记忆。
- **Phase I（Accumulation）**：多 rollout 总结 + 跨 rollout critique + 分层合并。  
- **Phase II（Inference）**：任务分解检索 + 视觉条件改写 + 注入执行提示。  
- 将使用历史回流到后续积累，形成持续改进回路。

## 4) Experiments（必须含具体数字）
### 4.1 Experimental setup
- **任务/数据：**5 个 benchmark，分三类域：
  - visual tool use：VisualToolBench、TIR-Bench
  - multimodal search：MMSearch-Plus、MMBrowseComp
  - comprehensive：AgentVista
- 每个 benchmark 随机采样 **100 tasks** 做 accumulation 训练，其余测试。  
- **模型/agent 配置：**Gemini-2.5-Pro、Gemini-3-Flash、GPT-5-mini、o4-mini。  
- **对比基线：**w/ tools（tool-only）、AWM、Dynamic CheatSheet（DC）、Agent-KB。  
- **评测指标：**N=4 rollouts，报告 **Average@4** 与 **Pass@4**。  
- **关键实现数字：**retrieval top-k=3；temperature=0.6；top-p=1.0；max turns=20；embedding=text-embedding-3-small。  
- **缺失标注：**`θ_sim`、`τ_min`、`L_max^e`、`L_max^K`、`N_max^E` 等阈值主文仅给符号，具体值在可见正文中**缺失**。

### 4.3 Analysis experiments（强制“现象+解释”）
- **现象：**去掉 experiences 掉 **3.04**，去掉 skills 掉 **3.85**（VisualToolBench, Gemini-2.5-Pro）。  
  **解释（作者）：**两条知识流互补，但 skills 对执行可靠性贡献更大。  
  **【标注】（我的判断，可选）：**对工具密集任务，先把“结构化可执行模板”做好，比只加经验片段更稳。

- **现象：**Phase I 组件去除降幅更大：Experience Manager **-4.09**、Skill Manager **-3.62**；Phase II 去除：Task Decomposition **-1.28**、Task Adaptation **-1.52**。  
  **解释（作者）：**知识库质量是上限，检索机制是放大器而非根因。  
  **【标注】（我的判断，可选）：**工程落地应优先投入写入链路（总结/批判/合并/去重），其次再做 retrieval 优化。

- **现象：**Error 率从 **29.9% (168)** 降到 **15.3% (95)**；syntax error **114→71**；tool-name error **16→2**。  
  **解释（作者）：**skills 提供模板与流程约束，减少结构性错误与无效恢复。  
  **【标注】（我的判断，可选）：**这说明 skill 文档实质是“可执行 SOP 约束层”，不是普通知识库说明文。

- **现象：**Experiences 改变工具调用分布：VisualToolBench 中 code interpreter **66.63%→74.49%**（full pipeline **76.97%**）；MMSearch-Plus 中 code interpreter **6.18%→13.21%**、image search **15.43%→24.63%**。  
  **解释（作者）：**experience 提供条件化战术，提升工具选择灵活性。  
  **【标注】（我的判断，可选）：**经验流主要优化“决策策略分布”，skill 流主要优化“执行稳定性”。

- **现象：**rollout N 从 1→4 时，Average@4 与 Pass@4 持续上升，且 Pass@4 增幅更陡。  
  **解释（作者）：**更多轨迹多样性使 critique 更易抽取可迁移因果模式。  
  **【标注】（我的判断，可选）：**可做预算感知采样：优先保留分歧大的 rollout，提高记忆学习效率。

## 5) Why it matters for our work
- 对 memory-agent 系统设计的直接启发：
  1) **双池记忆 schema**（experience/skill 分离）优于单池混存；
  2) **视觉对齐检索与改写**是多模态任务中的一等公民；
  3) **写入质量优先**（Phase I>Phase II）可作为资源分配原则；
  4) 跨模型迁移结果说明外置知识库具备模型解耦价值。
