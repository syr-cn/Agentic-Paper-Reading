## 1) One-line Why-read
ARISE 的关键价值是：把“选技能（before）—解题（during）—写入技能（after）”三步统一进同一个共享策略的 RL 闭环，让 policy gradient 直接塑造 skill selection，而不是依赖外挂 retriever。

---

## 2) CRGP
### C — Context
RLVR/GRPO 在数学推理后训练里有效，但“每题独立优化”导致经验难累积；已有 memory/skill 方法常把检索器放在策略外部。

### R — Related work
- **RLVR 线**：GRPO、Dr.GRPO、DAPO、GSPO。  
- **Memory/skill 线**：Reflexion / ExpeL / SimpleMem（偏推理时检索），EvolveR / SkillRL / SAGE（开始把 skill 放进训练环）。

### G — Gap
1. skill selection 通常由外部模块完成，policy gradient 无法端到端优化“选什么 skill”；  
2. skill 库更新与 RL 目标弱耦合，难形成“策略变强 ↔ 记忆更优”的正反馈。

### P — Proposal
ARISE 用 **Manager-Worker 共享参数策略** + **两层 skill 库（cache/reservoir）** + **分层奖励** + **两阶段训练** 解决上述问题：
- 奖励定义：\(R\in\{r_0,r_1,r_2\}\)，实验中设定 **r0=0, r1=1, r2=2**；
- 训练流程：Phase I warm-up（前 **500 steps**）→ Phase II skill-augmented（从 step 501 开始）；
- 每个 query rollout 组大小：**G=8**，并有额外 skill 生成 rollout \(O_{G+1}\)。

## 4) Experiments
### 4.1 Setup（带数字）
- **训练数据**：DeepScaleR，约 **40K** 题（AMC/AIME/MATH/OlympiadBench）。  
- **基座模型**：Qwen3-4B-Instruct-2507、Phi-4-mini-instruct-3.8B。  
- **训练步数**：**2000** optimization steps。  
- **硬件**：A100-80GB（原文写“× A100-80GB”，具体卡数在可提取片段中未明确）。  
- **GRPO 关键参数**：
  - group size **G=8**；
  - per-device batch size **4**；gradient accumulation **2**；
  - effective batch **64 queries/update**；
  - AdamW，lr **1e-6**，weight decay **0.01**；
  - cosine schedule + **50** warmup steps；
  - max generation length **4096 tokens**（含 \(O_1...O_G\) 与 \(O_{G+1}\)）；
  - clipping \(\epsilon_c=0.2\)。
- **两阶段**：
  - Phase I：前 **Nw=500** steps，binary reward \(r^{task}\in\{0,1\}\)；
  - Phase II：step **501+**，分层奖励 \(R\in\{0,1,2\}\)。
- **评测**：7 benchmarks（AMC23、AIME24、AIME25 + Omni-MATH 四子域），**Pass@1 平均 32 runs**。

### 4.3 Analysis（现象 + 解释 + 我的判断）
1) **现象**：去掉分层奖励（改 binary reward）时，准确率下降最大；skill utilization 从 **73% 降到 31%**。  
   **解释（作者）**：\(r_2>r_1>r_0\) 让“同样答对但是否有效用 skill”在 advantage 上可区分，驱动策略偏向可复用技能。  
   **我的判断**：这点非常关键，说明“记忆使用奖励”比“仅任务成功奖励”更能把 memory 内生化到策略里。

2) **现象**：随机注入 skill 时 utilization 高，但准确率反而下降。  
   **解释（作者）**：高使用率不等于高质量检索，不匹配 skill 会引入噪声上下文。  
   **我的判断**：对 agent memory 系统是直接警示：不能只追“召回率/调用率”，必须同时优化“匹配质量/有用性”。

3) **现象**：去掉 \(O_{G+1}\)（skill 生成 rollout）后，库冻结在 **24 seed skills**，且 OOD（Omni-MATH）退化最明显。  
   **解释（作者）**：静态启发式覆盖不了 Olympiad 级多样题型，库需要随训练增长。  
   **我的判断**：对长程 agent 同理——只靠冷启动 memory 不够，必须在线蒸馏新经验对象。

4) **现象**：去掉 confidence gate 影响最小，但 utilization 升到 **91%**。  
   **解释（作者）**：gate 主要是边界样本的噪声过滤。  
   **我的判断**：可把 gate 视为“memory 调用阈值控制器”，先保 precision 再调 recall。

5) **现象**：后期库规模趋于饱和，但准确率继续上升。  
   **解释（作者）**：后期收益来自“更会选已有 skill”，而非“继续堆库”。  
   **我的判断**：与“记忆治理 > 记忆堆量”一致，后期应更多投资源在路由/重排/淘汰机制。

## 5) Why it matters for my work（agent memory / long-context / multimodal RL）
- **对 agent memory**：ARISE 给出“memory 进入策略梯度”的可执行路径，不再是外接检索器。  
- **对 long-context**：两层库（活跃 cache + reservoir）天然对应“短期工作记忆 + 长期归档记忆”的分层治理。  
- **对 multimodal RL**：Manager-Worker 结构可迁移到“跨模态 skill 路由”，例如先选视觉/文本/工具 skill，再执行具体推理或操作。

## 6) Actionable next steps（3 条，可直接执行）
1. **在现有 RL pipeline 加入三值奖励**：把“任务成功”与“记忆是否实质贡献”拆开打分（0/1/2），并记录 skill-conditioned advantage 的分布变化。  
2. **实现两层记忆池 + 五操作最小版**：cache/reservoir + Add/Update/Evict/Load/Delete，先在长上下文任务上验证“库饱和后性能是否继续提升”。  
3. **做多模态对照实验**：external retriever vs policy-driven retrieval，在视觉问答或工具调用任务上比较（准确率、token 开销、memory 利用率、OOD 泛化）。
