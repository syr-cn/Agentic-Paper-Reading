## 1) 一句话 Why-read（必填）
- **Key claim/contribution + key observation：** DIVE 把任务合成顺序从 query-first 反转为 **evidence-first（先跑真实工具拿证据，再反推题目）**，在保证可执行/可验证的同时，把“结构多样性”做成可扩展主轴，显著提升跨工具/跨任务 OOD 泛化。

## 2) CRGP 拆解 Introduction（必填）
### C — Context
- 工具调用 agent 的训练越来越依赖合成任务，但很多数据只覆盖固定流程（如 search→browse），导致模型在新工具/新任务上迁移差。
- 真正上线场景需要跨域工具（金融、医疗、生物、软件等）和多样调用结构，而非单一套路。

### R — Related work
- **Pipeline extraction**：从专用系统抽数据，质量可控但扩域成本高。
- **Simulated tools**：可大规模生成，但可执行性与可验证性不稳。
- **Query-first on real tools**：先编题再验证，容易出现不可解题，人工质检成本高。

### G — Research gap
- 缺一个同时满足四点的数据合成方案：
  1) 结构多样；2) 可验证；3) 可执行；4) 可规模化。
- 现有方法通常在“多样性 vs 可靠性”上二选一。

### P — Proposal
- 提出 **Dive**：Evidence Collection → Task Derivation 的闭环，核心是执行优先（grounding by construction）。
- 资源池解耦：
  - **373 tools / 5 domains**（通用 + Finance/Biology/Medicine/Academia）
  - 每域约 **~5,000** seeds
  - query-only exemplars（每次采样 **3–5**）
- 配置采样与迭代：
  - 每题工具子集 **15–50**
  - 每轮 evidence collector 最多 **6** 次 tool-calls
  - collection-derivation **K=3**

## 4) Experiments（必须含具体数字）
### 4.1 Experimental setup
- **任务/数据：**
  - SFT 候选池 **114k tasks**，拒绝采样后保留 **48k trajectories**。
  - RL 候选池 **38k tasks**，按 pass@8 成功数 **1–5** 过滤得 **3.2k frontier tasks**。
- **模型/agent 配置：**
  - Backbone: **Qwen3-8B**。
  - 合成器：evidence collector 与 task generator 均为 **Claude-4-Sonnet**。
  - 合成超参：tool subset **15–50**，exemplars **3–5**，每轮最多 **6** 次工具调用，**K=3**。
- **训练设置：**
  - SFT：**300 steps**, batch **64**, lr **1e-5**, context **65,536**, max tool turns **50**。
  - RL（GRPO）：**100 steps**, batch **512**, lr **5e-6**, context **131,072**, max tool turns **100**。
- **对比基线：**WebExplorer-8B, SWE-Dev-8B, EnvScaler-8B；以及多家 >8B frontier 模型。
- **评测指标：**各 benchmark success rate（%），Toolathlon 报告 3-run 平均，其余多为 4-run 平均。
- **缺失标注（复现相关）：**
  - 训练硬件规模：**未披露**
  - 总训练 token / wall-clock：**未披露**
  - 工具调用成本与失败重试策略：**未完整披露**
  - 统计显著性检验（如 p-value / CI 完整表）：**未系统披露**

### 4.3 Analysis experiments（强制“现象+解释”）
- **现象1：** Diversity-only（12k 固定、扩域 1→4）总体优于 Quantity-only（固定分布、12k→48k），且在多数 OOD 上“少 4× 数据仍更强”。
  **解释（作者）：** 泛化瓶颈在结构多样性不足，不在样本总量不足。
  **【标注】（我的判断）：** 结论可信且可迁移到我们数据工程：先扩工具能力边界，再扩同分布样本量。

- **现象2：** Toolset-variety-only 会较快饱和；Pool-Expansion+Variety 上升更快、上限更高。
  **解释（作者）：** 两者的“组合数增长”相近，核心差异在是否引入新工具类型/新能力。
  **【标注】（我的判断）：** “随机重排旧工具”不等于真正多样性；能力底座宽度才是关键变量。

- **现象3：** RL 在高多样性数据上收益更大（文中给出 AvgRL–AvgSFT 从 +4.6 增至 +5.6）。
  **解释（作者）：** RL 不仅模仿轨迹，还在多样环境中探索并强化更广的有效策略结构。
  **【标注】（我的判断）：** 与我们经验一致：环境多样时，RL 才更像“能力放大器”而非“模板微调器”。

- **现象4：** 48k SFT 对比 Gen-DR，结构多样性大幅提升：tools covered **2→373**、unique toolsets **1→46,398**、topology classes **65→153**。
  **解释（作者）：** evidence-driven 迭代产生更丰富 R/P 结构（PureR→R+P→PureP）。
  **【标注】（我的判断）：** 指标体系有价值；但 tool-call graph 由 LLM 推断，仍建议补程序化抽取做交叉验证。

## 5) Why it matters for our work
- 对 memory agent：应把“可回放工具证据链”作为记忆原子，而不只存最终 QA。
- 对 long-context 训练：65k/131k context + verifiable targets 的组合可行，说明超长上下文在工具轨迹监督中有实际收益。
- 对跨域 agent：DIVE 显示“能力覆盖率”比“同构数据量”更影响 OOD 上限。

### Case 对齐（>=2）
- **Case A（对齐 `2026-03-11_evo-memory.md`）**：都强调“结构信号”比“表面文本量”更关键。DIVE 把结构信号具体化为可执行 tool-use topology。
