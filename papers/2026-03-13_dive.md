## 1 一句话 Why-read
这篇最值得读的核心是：把 agent 任务合成顺序彻底反过来——**先执行真实工具拿证据，再反推任务**，从机制上同时保证可执行/可验证，并把“多样性”作为主 scaling 轴来提升 OOD 工具泛化。

---

## 2 CRGP

### C — Context
- 现有 agent 训练很依赖合成数据，但常见做法要么是：
  1) 固定任务族 + 固定工具集（导致策略僵化，如 search→browse 循环），
  2) 或模拟工具环境（可扩展但容易不稳定、不可验证）。
- 论文认为关键瓶颈不是“数据不够多”，而是“**结构多样性不够**”：工具类型覆盖、每题工具子集组合、以及多步工具调用形态都不够丰富。

### R — Related work
- query-first 合成（先造问题再验可做）在真实工具环境下验证成本高，且容易出现不可解任务。
- simulated tools 路线扩展快，但执行稳定性与真实性不足。
- pipeline extraction 路线（从专用系统抽数据）工程成本高，扩域慢。

### G — Research gap
- 缺一个能**同时**满足四点的数据合成方案：
  1) 结构多样（structurally diverse）
  2) 可验证（verifiable）
  3) 可执行（executable）
  4) 可规模化（scalable）
- 现有方法通常在“多样性 vs 可靠性”之间二选一，无法兼得。

### P — Proposal
- 提出 **Dive**：evidence-first（执行优先）任务合成。
- 资源池三件套（解耦采样）：
  1) **373 个真实工具**，覆盖 **5 个域**（通用 + Finance/Biology/Medicine/Academia）
  2) seed pool：每域约 **~5,000** 实体种子
  3) exemplar pool：query-only 样例（3–5 个/次）
- 每轮配置采样：
  - 工具子集大小 **15–50**
  - exemplar **3–5**
  - 迭代 **K=3**（收集证据→反推 QA）
- 训练：
  - SFT：**48k**（来自 114k 任务池，经拒绝采样）
  - RL：**3.2k** frontier tasks（从 38k 任务池筛出）
- 主结论：Qwen3-8B 上，跨 OOD 基准平均提升显著（文中报告 **+22.2**，RL）。

## 4 Experiments

### 4.1 Experimental setup（具体数字/设置）

#### 合成阶段
- evidence collector / task generator：均用 **Claude-4-Sonnet**。
- 每轮采样：seed + 工具子集（**15–50**）+ exemplars（**3–5**）。
- evidence collector 每次迭代最多 **6** 次 tool-calling。
- collection-derivation 迭代次数：**K=3**。
- 工具执行：**live tools**（非模拟）。

#### 训练阶段（Qwen3-8B）
- **SFT**：
  - 任务池 **114k**，教师 **GPT-OSS-120B**
  - 留存 **48k** 轨迹
  - 训练 **300 steps**，batch **64**，lr **1e-5**
  - max context **65,536** tokens
  - 最多 **50** tool-call turns
- **RL（GRPO）**：
  - 从独立 **38k** 任务池筛 frontier tasks
  - 筛选规则：pass@8 自采样成功数在 **1–5**
  - 最终 **3.2k** 任务
  - 训练 **100 steps**，batch **512**，lr **5e-6**
  - max context **131,072** tokens
  - 最多 **100** tool-call turns

#### 评测
- 三层 benchmark：L1（ID）+ L2（一般工具 OOD）+ L3（专用工具 OOD）
- 文中主表覆盖 GAIA/HLE/BrowseComp/Xbench-DS/FinSearchComp/FAB/MAB/SWE-bench Verified/Toolathlon 等。
- 解码设置：temperature=1，top-p=1。

> 若你追求完全可复现：
> - 训练硬件、总 token/总 wall-clock、工具调用成本等，**原文未给出可提取数字**。

### 4.3 Analysis（至少 3 条：现象+解释+我的判断）

1) **现象：** 在匹配预算下，diversity-only（12k，扩域）优于 quantity-only（48k，固定分布），即使数据量少 4× 仍更强。  
   **解释（作者）：** OOD 泛化瓶颈在结构多样性，不在样本总量；固定流程会强化套路而非能力。  
   **我的判断：** 这个结论非常可信，且对 agent 数据工程有直接指导价值：先扩工具类型和组合空间，再谈“灌更多同分布轨迹”。

2) **现象：** “仅增加 toolset variety（池不扩）”会较快饱和；“pool expansion + variety”提升更快且天花板更高。  
   **解释（作者）：** 两条路径的 toolset 变体增长相近，真正差异在是否引入新工具能力类型。  
   **我的判断：** 这点很关键：多样性不是“随机组合更多同类工具”，而是“能力基底真正变宽”。对 memory agent 来说，对应的是“技能库覆盖率”优先级高于“同技能重采样”。

3) **现象：** RL 对高多样性数据的增益更大（1 域时 RL-SFT gap +4.6；4 域时 +5.6），且 4 域平均分 29.1→34.8。  
   **解释（作者）：** RL 不只是拟合 teacher 轨迹，而是在多样环境中探索并固化更多有效结构。  
   **我的判断：** 我认可这个机制解释。多样数据提高了“可探索状态空间”的价值密度，RL 才不容易坍缩到单一路径。

4) **现象：** 48k SFT 下，Dive 的结构多样性统计远高于 Gen-DR：
   - tools covered: 2→373
   - unique toolsets: 1→46,398
   - unique call sequences: 1,231→25,084
   - topology classes: 65→153  
   **解释（作者）：** 证据驱动循环诱导出更丰富 R/P（Retrieval/Processing）拓扑。  
   **我的判断：** 指标体系设计得不错，至少避免只看最终分数；但图结构提取依赖 LLM 判别（Claude-4-Sonnet），可能引入评估器偏差，建议后续用程序化图抽取复核。

## 5 Why it matters for our work
- 对 **agent memory**：Dive 说明“可回放证据链”是高质量记忆单元的天然形式，不应只存 QA 文本结果。
- 对 **long-context**：SFT/RL 已在 65k/131k 上跑通，说明超长上下文在工具轨迹监督中确实可用，但要配套 verifiable target 才稳。
- 对 **multimodal RL**：其核心思想可迁移到图像/表格/时序工具——先拿真实模态证据，再反推任务，降低“虚构题面”。

## 6 Actionable next step（仅给 3 条，可执行）

1) **落地 evidence-first 记忆写入链路（agent memory）**  
   - 实施：在现有 agent 中新增 `evidence_trace` schema（tool, args, obs, timestamp, verifier）。
   - 1 周内产出：至少 1k 条可回放 trace；构建“不可回放率”监控（目标 <5%）。

2) **做 diversity-vs-quantity 对照实验（long-context）**  
   - 实施：固定训练 token 预算，A 组扩样本数量，B 组扩工具池/工具组合/任务拓扑。
   - 2 周内产出：在 2 个 OOD 集上比较增益；若 B 组显著优于 A 组，则将数据工程优先级切换到 diversity scaling。

3) **做 multimodal RL 的“证据先行”小规模试点**  
   - 实施：选一个图像检索+表格计算混合任务，先跑真实模态工具链，再自动反推 query-answer。
   - 2–3 周内产出：构建 500–1,000 条可验证样本；用 GRPO 小步训练，检查 RL 是否同样放大多样性收益。

## 7 评分解释（维持原倾向）
- **维持 4/5，不上调**：
  1) 方法论非常实用，且有扎实数字支持（多层 OOD + scaling 分析 + 结构多样性统计）。
  2) 但仍有一些关键信息未充分公开（如完整成本/硬件/更细粒度方差与显著性细节），工程复现难度不低。
- 结论：这是一篇“可直接指导 agent 数据工程策略”的强 paper，但尚未到“几乎无短板”的 5 星级别。
