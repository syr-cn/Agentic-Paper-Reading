## 1) One-line Why-read
这篇最值得读的点：**把 memory operation 从“固定规则流水线”升级成“可学习、可演化、可复用的 skill bank”**，并用 controller/executor/designer 做闭环优化。

---

## 2) CRGP

### C — Context
长程 agent 越来越依赖 memory，但主流 memory 系统多是 hand-crafted：固定 add/update/delete/skip + 模块化启发式，遇到分布变化和长历史时鲁棒性差。

### R — Related Work
- 记忆系统：MemoryBank、A-MEM、Mem0、MemoryOS、ReadAgent、CoN
- RL 化 memory 管理：Memory-R1、Mem-α
- 自演化 agent：ExpeL、EvolveR、AgentEvolver、RAGEN、SkillWeaver
- 与并行工作区别：Evo-Memory 更偏 benchmark；MemEvolve 更偏架构空间元优化；**MemSkill 聚焦“技能本身的演化”**。

### G — Gap
作者要补的核心缺口：
1. **人类先验过重**：固定操作模板不够自适应；
2. **提取粒度僵硬**：很多方法绑定 turn-level，扩到长 span 性能/成本不理想；
3. **缺少可组合技能条件化**：现有往往是静态模块流水线，不是按上下文动态组合技能。

### P — Proposal
MemSkill 三件套：
- **Controller**：按当前 span + 已检索记忆选 Top-K skills；
- **Executor (LLM)**：一次生成结构化 memory update；
- **Designer (LLM)**：周期性看 hard cases，精修旧技能并新增技能。

关键机制与数字：
- 初始 skill bank = **4 原语**（Insert/Update/Delete/Skip）
- 训练期 conversational：**K=3**
- 评测期：LoCoMo/LongMemEval **K=7**，ALFWorld **K=5**
- 记忆检索上限：**20 items**
- designer 触发：每 **100 steps**
- 每轮 skill edit 上限：**3**
- ALFWorld 最长交互：**50 steps**
- eval span/chunk size：**512**

## 4) Experiments

### 4.1 Setup（可核对数字）
- 数据：LoCoMo、LongMemEval、ALFWorld、HotpotQA（迁移）
- 指标：
  - 对话：F1、L-J（LLM Judge）
  - 交互：SR、#Steps（越低越好）
- Base LLM：LLaMA-3.3-70B-Instruct；Qwen3-Next-80B-A3B-Instruct
- 编码/检索：Qwen3-Embedding-0.6B；Contriever
- 统一 retrieval cap：**20**
- 训练：PPO 初始化 controller
- 关键超参：K（train=3；eval: 7/5）、designer=100 steps、max edits=3、chunk=512、ALF max step=50

### 4.2 Main Results（Table 1）

**LLaMA-3.3-70B block：**
- Conversational Avg (L-J)：A-MEM 42.23 → **MemSkill 55.19（+12.96）**
- ALF Avg SR：LangMem 36.47 → **MemSkill 47.44（+10.97）**
- 细项：
  - LoCoMo：F1 **38.78**，L-J **50.96**
  - LongMemEval：F1 **31.65**，L-J **59.41**
  - ALF-Seen：SR **47.86**，#Steps **30.88**
  - ALF-Unseen：SR **47.01**，#Steps **30.43**

**Qwen3-Next-80B block（transfer eval）：**
- Conversational Avg (L-J)：A-MEM 41.53 → **MemSkill 55.99（+14.46）**
- ALF Avg SR：CoN 55.80 → **MemSkill 62.09（+6.29）**
- 细项：
  - LoCoMo：F1 **39.28**，L-J **52.07**
  - LongMemEval：F1 **23.75**，L-J **59.90**
  - ALF-Seen：SR **60.00**，#Steps **24.54**
  - ALF-Unseen：SR **64.18**，#Steps **23.57**

### 4.3 Ablation（Table 2, LoCoMo L-J）

| Variant | LLaMA | Δ vs default | Qwen | Δ vs default |
|---|---:|---:|---:|---:|
| MemSkill default | **50.96** | - | **52.07** | - |
| w/o controller (random skills) | 45.86 | -5.10 | 41.24 | -10.83 |
| w/o designer (static skills) | 44.11 | -6.85 | 34.71 | -17.36 |
| Refine-only (no new skills) | 44.90 | -6.06 | 46.97 | -5.10 |

结论：controller 和 designer 都是必要组件，且 designer 的贡献更“硬”。

### 4.4 Case（>=2）

**Case 1（LoCoMo 型）**
- 任务形态：多轮对话、时间跨度长、人物事件交错。
- 演化后技能偏好：更强调“谁-何时-做了什么-在哪里”的事件组织与时间线连贯。
- 实际意义：减少后续问答时的跨轮拼接错误，L-J 提升更明显。

**Case 2（ALFWorld 型）**
- 任务形态：交互式决策，需要可执行状态记忆。
- 演化后技能偏好：对象位置、动作前置条件、任务约束（不是泛摘要）。
- 实际意义：在 Seen/Unseen 都有 SR 增益（如 Qwen Unseen SR **64.18**），说明“可执行记忆”确实影响长程行为成功率。

## 5) Analysis（>=3）

1. **现象：去掉 designer 退化最大**（Qwen: 52.07 → 34.71，-17.36）。
   - 解释：固定 4 原语不足以覆盖复杂失败模式，必须让 skill bank 继续生长。
   - 我的判断：这说明“可扩展技能空间”优先级高于“只优化检索/打分器”。

2. **现象：Qwen 上 LongMemEval 的 F1 不最强（23.75），但 L-J 最强（59.90）。**
   - 解释：技能迁移改善回答可用性与结构化质量，Judge 更敏感。
   - 我的判断：存在 metric mismatch 风险；高风险应用要加可验证指标（EM/证据一致性）。

3. **现象：Refine-only 明显优于 static skills，但仍低于 full。**
   - 解释：仅“修旧”无法覆盖新错误类型，“增新技能”提供新行为原语。
   - 我的判断：后续必须做 skill lifecycle（merge/prune/retire），否则库会膨胀并伤害选择稳定性。

4. **现象：HotpotQA 迁移里 K 越大通常越好，K=7 最优。**
   - 解释：长且噪声高上下文需要更多技能组合。
   - 我的判断：可做 context-adaptive K（按不确定性动态选 K），在不降质量下减 token 成本。

## 6) Why it matters / Next actions

1. **两周最小复现**：4 原语 + Top-K controller + 单次 executor，先对齐 retrieval cap=20，跑 A/B。  
2. **自适应 K**：比较固定 K=7 vs 动态 K，联合看 L-J/F1 与 token/latency。  
3. **designer 安全阀**：保留论文思路（rollback + early stop + edit budget=3），再加 skill 去重与版本管理。
