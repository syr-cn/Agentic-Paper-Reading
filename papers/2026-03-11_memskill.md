## 1) One-line Why-read
MemSkill 的核心价值在于：**把 memory operation 从“固定流程”升级为“可学习 + 可演化的 skill bank”**，并用 controller/executor/designer 形成闭环，这比单纯改检索器更系统。

---

## 2) CRGP

### C — Context
LLM agent 在长程交互中依赖 memory，但主流方法仍以手工固定操作（如 add/update/delete/skip）和静态启发式为主，适配性差、长历史扩展性有限。

### R — Related work
- Memory 管线类：MemoryBank、A-MEM、Mem0、MemoryOS、ReadAgent、CoN。
- RL 优化 memory 管理：Memory-R1、Mem-α。
- Self-evolving agent 相关：ExpeL、EvolveR、AgentEvolver、RAGEN、SkillWeaver。
- 论文定位：不同于“演化架构”或“测试时记忆更新 benchmark”，MemSkill专注于**演化 memory skills 本身**。

### G — Research gap
作者明确补的缺口有三点：
1. **人类先验过重**：固定 memory 规则难适配任务变化；
2. **提取粒度受限**：很多方法绑定 turn-level，难处理更长跨度；
3. **缺少可组合 skill 条件化构建**：现有是模块流水线，不是按上下文动态组合技能。

### P — Proposal
MemSkill 用三角色闭环：
- **Controller**：按当前 span + 检索记忆选择 Top-K skills；
- **Executor**（LLM）：一次性按技能条件生成 memory update；
- **Designer**（LLM）：周期性读取 hard cases，改写旧技能、增加新技能。

关键机制与数字：
- 初始 skill bank：**4 个原语**（Insert / Update / Delete / Skip）；
- 训练时 conversational 任务选择 **K=3**；
- 评测时 LoCoMo/LongMemEval 用 **K=7**，ALFWorld 用 **K=5**；
- 技能演化触发频率：**每 100 training steps**；
- 每轮最多 **3 次 skill edits**；
- 检索 memory 条目上限：**20**（和 baselines 保持一致）。

## 4) Experiments

### 4.1 Experimental setup（含可提取数字）
- **数据集**：LoCoMo、LongMemEval、ALFWorld、HotpotQA（分布外迁移分析）。
- **指标**：
  - 对话任务：F1 + L-J（LLM Judge）
  - ALFWorld：SR + #Steps（越低越好）
- **基座模型**：
  - LLaMA-3.3-70B-Instruct
  - Qwen3-Next-80B-A3B-Instruct
- **表示与检索**：
  - state/skill 编码：Qwen3-Embedding-0.6B
  - retriever：Contriever
  - memory retrieval 上限：20
- **训练/推理关键设置**：
  - 训练（对话任务）Top-K = 3
  - 评测 Top-K：LoCoMo/LongMemEval = 7；ALFWorld = 5
  - span/chunk size（eval）= 512
  - designer trigger = every 100 steps
  - max skill edits / evolution round = 3
  - ALFWorld max env interaction length = 50 steps

细粒度例子：
- LLaMA/LoCoMo: F1 **38.78**（略高于 MemoryOS 38.68），L-J **50.96**（显著高于 A-MEM 46.34）
- LLaMA/LongMemEval: F1 **31.65**，L-J **59.41**（该列最佳）
- Qwen/LongMemEval: F1 **23.75** 并非最佳，但 L-J **59.90** 为最佳（显示“事实精确率 vs judge感知质量”差异）
- Qwen/ALF-Unseen: SR **64.18**，#Steps **23.57**

#### (B) 消融（Table 2, LoCoMo L-J）

| Variant | LLaMA | vs default | Qwen | vs default |
|---|---:|---:|---:|---:|
| MemSkill default | **50.96** | - | **52.07** | - |
| w/o controller (random skills) | 45.86 | -5.10 | 41.24 | -10.83 |

结论很直接：controller 与 designer 都是刚需，且 **designer 缺失损失更大**（尤其 Qwen）。

### 4.3 Analysis（至少 3 条：现象 + 解释 + 我的判断）

1) **现象：**去掉 designer 的退化幅度最大（Qwen: 52.07→34.71，-17.36）。  
   **解释（作者）：**仅靠固定 4 原语技能集，难覆盖真实分布中多样失败模式；需要从 hard cases 持续补技能。  
   **我的判断：**这个结果很有说服力，说明“技能空间可扩展性”比“仅优化选择器”更关键；未来如果做长上下文 agent memory，优先投入 skill evolution 而不是先堆更复杂检索器。

2) **现象：**Qwen 迁移设置下，LongMemEval 的 F1 不顶尖（23.75），但 L-J 却是最佳（59.90）。  
   **解释（作者）：**技能迁移带来更高质量的记忆组织与回答可用性，L-J 捕捉到“整体回答有用性”。  
   **我的判断：**这里存在指标错位风险：judge-friendly 不等于事实最准确。若用于高风险任务，应加入可验证指标（EM/证据一致性）做双阈值。

3) **现象：**Refine-only 明显优于 static skills，但仍弱于 full MemSkill（LLaMA: 44.90 < 50.96；Qwen: 46.97 < 52.07）。  
   **解释（作者）：**只改写旧技能可修补局部问题，但无法覆盖新型失败模式；“新增技能”提供新的操作语义。  
   **我的判断：**这提示 skill bank 应该允许“生长”，但要配套压缩/去重，否则长期会膨胀、选择噪声升高。可考虑 skill merge/prune 的生命周期管理。

4) **现象：**HotpotQA 迁移中，随着上下文增大（50/100/200 docs），更大的 K 更优，K=7 最好。  
   **解释（作者）：**长且噪声高的上下文需要多技能组合来覆盖不同记忆行为。  
   **我的判断：**这说明 K 不是常数超参，应该做 context-adaptive K（按熵/不确定性动态调节），可减少短上下文时的冗余技能调用开销。

## 5) Why it matters for our work
- 对 **agent memory**：给了一个可落地的“技能化 memory operator”范式，且有 RL + evolution 闭环，不再只靠 hand-crafted 规则。
- 对 **long-context**：span-level 一次生成 + Top-K skills 的做法，比 turn-level 多次处理更有扩展潜力。
- 对 **multimodal RL**：designer 的 hard-case 驱动演化可迁移到视觉/行动轨迹记忆（把“失败轨迹”转成新 memory skill）。

## 6) Actionable next steps（3 条，可执行）

1. **Agent memory 原型复现（2 周）**  
   - 先实现最小闭环：4 初始技能 + controller Top-K + executor 一次生成；
   - 固定检索 top-20，与当前系统做 A/B；
   - 关键对比：K=3/5/7 对 latency、token cost、任务指标的影响。

2. **Long-context 自适应 K 实验（1~2 周）**  
   - 用上下文长度、检索分数熵、历史失败率做 gating，动态预测 K；
   - 对比固定 K=7，目标是在不降效果下减少技能注入 token；
   - 指标：质量（L-J/F1）+ 成本（tokens/call、总延迟）。

3. **Multimodal RL 的 designer 扩展（3 周）**  
   - 在图文/环境轨迹任务中构建 hard-case buffer（含视觉证据、动作序列、失败标签）；
   - designer 每 N step（可先 N=100）做技能编辑并限制每轮 edit budget（先设 3）；
   - 新增“回滚 + 早停 + skill 去重”机制，防止技能库失控增长。
