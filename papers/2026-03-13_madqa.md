## 1) 一句话 Why-read（必填）
- **Key claim/contribution + key observation：** MADQA 的关键不只是“再加一个 DocQA 数据集”，而是把评测从单点 Accuracy 推进到 **Accuracy × Effort Calibration（Kuiper）**，直接暴露“高分但低效率”的 agent 失配问题。

---

## 2) CRGP 拆解 Introduction（必填）
### C — Context
- 多文档 QA 基准常只看最终答对率，但真实 agent 系统更关心“用多少步、何时停、是否花冤枉算力”。

### R — Related work
- 文中对照了 DocVQA、MP-DocVQA、MMLongBench-Doc、MuRAR、M3DocRAG 等文档理解/检索评测。  
- 同时评测了多类系统：Managed File Search、BM25 MLLM Agent、RLM、MDocAgent、HEAVEN、ColBERTv2+Llama。

### G — Research gap
1. 缺少区分“策略导航 vs 随机试错”的统一指标；  
2. 缺少把 accuracy 与 effort 绑定的评估协议；  
3. 缺少 construct validity 证明“不是词匹配/记忆污染就能刷分”。

### P — Proposal
- 提出 **MADQA（2,250 问题 / 800 PDF）**；  
- 采用 CTT（Classical Test Theory）做 split，保留难题哨兵池；  
- 三类主指标：**Accuracy、Attribution（Page F1/Doc F1）、Kuiper（effort calibration）**。

## 4) Experiments（必须含具体数字）
### 4.1 Experimental setup
- **任务/数据：**
  - 总规模：**2,250 QA / 800 docs / 18,619 pages / 12.2M tokens**。  
  - 文档长度：每文档页数 **median 5 / mean 23.3 / max 859**。  
  - 跳数：**82.7% single-hop**，**17.3% multi-hop**（X-Page **8.3%**，X-Doc **9.0%**）。
- **划分（CTT）：**
  - Test **500**，Dev **200**，Train **1,550**。  
  - 另有 20% test 预算用于“Sentinel Pool”（当前模型 0% 基线命中，保留长期 headroom）。
- **标注与质控：**
  - 标注筛选先给 **20 annotators** 同题试标，仅保留人工复核零错误者。  
  - 终版修订：替换 <1%（多为 typo）、扩展 gold <5%（可选上下文补全）。
- **评测指标：**
  - Accuracy（LLM judge，去除约 50% exact-match 后，与人工一致性 **quadratic-weighted Cohen’s κ=0.88**）；  
  - Attribution（Page F1 / Doc F1）；  
  - Calibration（Kuiper，越低越好）；  
  - 努力度量鲁棒性：step 与 token 指标相关 **Spearman ρ>0.85**。
- **预算与实现细节：**
  - 常见步数上限：**N=10（typical）**。  
  - 各商用 API 的具体系统 prompt / 工具参数细项：主文未完整展开，需附录 **[缺失]**。  
  - 全系统统一硬件不可比（多为 API 调用），单一硬件统计 **[缺失]**。

### 4.3 Analysis experiments（强制“现象+解释”）
- **现象1：** 最优 agent（82.2）与 Human BM25 几乎同分，但与 Human Oracle 仍有 **17.2** 点差距。  
  **解释（作者）：** 当前上限主要受 retrieval/导航链路约束，不是单纯答案生成问题。  
  **【标注】（我的判断）：** 这说明后续优化优先级应是“前段路由+停止策略”，而非再堆 reader。

- **现象2：** 冷启动差异很大：人类首步约 **50%**，Gemini 3 Pro 首步约 **12%**。  
  **解释（作者）：** agent 依赖后续多步恢复（compute-intensive recovery）追平总体准确率。  
  **【标注】（我的判断）：** 在线上 latency/cost 受限场景，这种“后程追回”会显著掉队。

- **现象3：** Accuracy 与 Kuiper 不单调（77.7/52.6 可能比 82.2/25.8 校准差很多）。  
  **解释（作者）：** “答对能力”和“是否把预算花对地方”是两条独立轴。  
  **【标注】（我的判断）：** RL 奖励若只盯最终正确率，会诱导 brute-force；需显式惩罚长失败轨迹。

- **现象4：** 词匹配几乎不可解：unigram 精度 **0.03%**，bigram **2.6%**，trigram 召回仅 **51%**。  
  **解释（作者）：** 数据构造有效避免 lexical shortcut。  
  **【标注】（我的判断）：** 对 memory-agent 很友好，能较真实地测“检索+整合”而非 pattern hack。

#### Case（>=2）
- **Case 1（同准确率，不同能力结构）：Human BM25 vs Gemini 3 Pro BM25**  
  - 现象：二者都在 ~82.2 accuracy，但 item agreement 低（文中给 **Cohen’s κ=0.24**）。  
  - 解释：人机擅长题目簇不同（互补性强），并非同一种解题机制。  
  - 启示：可做 hybrid pipeline（人机协作分流）冲破单系统天花板。

- **Case 2（同家族高分模型，校准分化）：Gemini 3 Pro BM25 vs GPT-5 BM25**  
  - 现象：Gemini 3 Pro 在 accuracy 更高（82.2 vs 77.7）且 Kuiper 显著更低（25.8 vs 52.6）。  
  - 解释：后者在困难题上投入更多但回报低，effort-accuracy 对齐差。  
  - 启示：部署选型不能只看 leaderboard accuracy，需加预算曲线约束。

## 5) Why it matters for our work
- 对 **agent memory**：memory 的目标应是减少无效轮次，而非只提高“最终答对一次”。  
- 对 **long-context**：长上下文能力必须与导航策略绑定评估（Accuracy@N + Kuiper），否则看不出冷启动弱点。  
- 对 **multimodal RL**：57%+ 题目涉及结构/视觉信息，纯文本策略会遇到明确上限。
