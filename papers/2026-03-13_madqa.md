## 1) 一句话 Why-read
MADQA 的核心价值不是再加一个 DocQA 数据集，而是把评估目标从“只看答对率”推进到“**准确率 × 代价校准（effort calibration）**”，直接揭露 agent 在长文档检索中的“高分但低效率”问题。

---

## 2) CRGP
### C — Context
现有多文档/多页 QA 基准大多关注最终 accuracy，但真实 agent 任务里，**是否用合理步数找到证据并停止**同样关键。MADQA 把这件事制度化。

### R — Related work
- 论文对比了 DocVQA / MP-DocVQA / MMLongBench-Doc / MuRAR / M3DocRAG 等，指出既有任务常见局限：
  1) 单文档或弱 agentic 设定；
  2) 题目并非全人工；
  3) 对“过程效率”缺少可比指标。  
- 本文也纳入了大量强基线：Managed File Search、BM25 MLLM Agent、RLM、MDocAgent、HEAVEN、ColBERTv2+Llama 等。

### G — Research gap
作者要填的 gap 很明确：
1. **能否区分“策略导航”与“随机试错”**（同样准确率下，路径质量不同）。  
2. 缺少面向 agent 过程的联合评估：accuracy 与 effort 没有被统一衡量。  
3. 缺少可证明“不是词匹配/参数记忆就能过”的构造化验证（construct validity）。

### P — Proposal
- 发布 **MADQA：2,250 人工问题 + 800 异构 PDF**。  
- 用 Classical Test Theory（CTT）做 split 设计，维持区分度与长期挑战性。  
- 评估协议包含：
  - **Answer correctness**（LLM judge，校准后 κ=0.88）；
  - **Attribution**（Page F1 / Doc F1）；
  - **Effort calibration**（Kuiper，越低越好）。

## 3) Figure 区（关键图结论）

## 4) Experiments
### 4.1 Experimental setup（具体数字/设置）
#### 数据与任务
- 总规模：**2,250 questions / 800 docs / 18,619 pages / 12.2M tokens**。  
- 文档长度：每文档页数 **median 5 / mean 23.3 / max 859**。  
- 跳数分布：**17.3% multi-hop**（X-Page **8.3%**，X-Doc **9.0%**），其余 82.7% single-hop。  
- 领域：13 个高层 domain（论文给出各域文档与问题数）。

#### 划分
- Test: **500**（只放输入，标签隐藏用于 leaderboard）  
- Dev: **200**（公开 GT）  
- Train: 论文写“公开训练注释用于 RL/prompt-search”；按总量推算约 **1,550**（=2250-500-200，推导值）。

#### 标注与质控
- 先让 **20 annotators** 同组试标，双阶段筛选，仅保留人工复核零错误者。  
- 后处理修订：替换 <1%（多为 typo），扩展 gold <5%（多为可选上下文）。

#### 指标与评估协议
- Accuracy：LLM judge，排除约 50% exact-match 样本后，与人工一致性 **quadratic-weighted Cohen’s κ=0.88**。  
- Attribution：Page F1 / Doc F1。  
- Effort calibration：Kuiper（低更好），默认 effort=step/tool-calls。  
- 努力度量鲁棒性：steps 与 token 指标 Spearman **ρ>0.85**（正文），附录里多系统在 **0.72~0.95**。

#### 预算设置
- 评估常用上限步数：**N=10（typically）**。  
- 文中明确：模型排名会受预算影响，N≤3 与收敛态排名可能反转。

> 若你要复现实验级细节（如每个 baseline 的检索器参数、prompt 全文、工具超参），**主文未给出可提取数字**，需查 Appendix G。

### 4.2 Main result table（主结果+可解释 delta）
| Setting | Baseline | Proposed / Best | Delta |
|---|---:|---:|---:|
| **Overall Accuracy（最佳非 agentic vs 最佳 agentic）** | 78.6±2.2（Gemini 3 Pro File Search） | 82.2±2.0（Gemini 3 Pro BM25 Agent） | **+3.6** |
| **Oracle gap（相对 Human Oracle Retriever）** | 82.2（best agentic） | 99.4±0.4（Human Oracle） | **-17.2**（agent 未弥合） |
| **Effort calibration（Kuiper，越低越好）** | 25.8（best-acc agent） | 14.6（Human BM25） | agent **+11.2**（更差） |
| **Calibration non-monotonic 示例** | GPT-5 BM25: Acc 77.7, Kuiper 52.6 | Gemini 3 Pro BM25: Acc 82.2, Kuiper 25.8 | 高准确率不等于好校准 |

补充：
- 非 agentic Top-1 在 Accuracy 上已很强（78.6），但 Page F1/Doc F1 与 agentic 的差异说明“找对文档 ≠ 定位对页面”。  
- 论文结论“约 18% oracle gap”与具体数字 17.2%一致（四舍五入语义）。

### 4.3 Analysis（现象 + 解释 + 我的判断）
1. **现象：** 最优 agent 在准确率上可与人类 BM25 持平（82.2），但离 Human Oracle 仍有 **17.2%** 缺口。  
   **解释（作者）：** 主要瓶颈仍在 retrieval/导航效率，而非纯答案生成。  
   **我的判断：** 这是典型“检索链条瓶颈”信号：在多文档任务里，reader 再强也救不了错误上下文，后续应优先做 early-stage routing 与停止策略。

2. **现象：** 冷启动下人类首步约 **50%**，Gemini 3 Pro 首步约 **12%**，但大预算后 agent 可追平。  
   **解释（作者）：** agent 依赖“高成本恢复曲线”，通过更多试探把错误覆盖掉。  
   **我的判断：** 这不是“策略优”，而是“预算堆出来的鲁棒性”；产品化场景（延迟/成本约束）下会被放大成严重缺陷。

3. **现象：** Kuiper 与 Accuracy 不单调（77.7 accuracy 的系统可能比 82.2 accuracy 系统校准更差很多：52.6 vs 25.8）。  
   **解释（作者）：** 回答正确率和“是否把预算花在值得问题上”是两条正交轴。  
   **我的判断：** 这对 RL 奖励函数设计很关键：只奖正确率会鼓励 brute-force；需把“失败长轨迹惩罚”显式加进奖励。

4. **现象：** 构造效度验证显示词匹配策略基本不可行：unigram 精度 **0.03%**、bigram **2.6%**、trigram 召回仅 **51%**。  
   **解释（作者）：** 数据确实迫使系统做语义检索与跨页聚合。  
   **我的判断：** 这使 MADQA 对 long-context + memory agent 更有价值，不容易被“Ctrl+F 式 hack”刷分。

## 5) Why it matters for our work
对 agent memory / long-context / multimodal RL 的直接启发：
1. **评估层面**：必须从“单点正确率”升级为“正确率 + 过程校准（Kuiper/预算曲线）”。  
2. **系统层面**：memory 模块要服务“减少无效回合”，不是只做被动缓存。  
3. **训练层面**：RL 或策略搜索要惩罚 loop 和无收益扩展，而不是鼓励无限试探。  
4. **多模态层面**：57%+ 题目依赖结构/视觉理解，纯文本管线上限明显。

## 6) Actionable next steps（面向 agent memory / long-context / multimodal RL）
1. **做一个 MADQA-style 过程日志协议（1周内可落地）**  
   - 每题记录：step 数、token、检索命中率、Page/Doc 对齐、终止原因。  
   - 同时输出 Accuracy@N 曲线（N=1..10）与 Kuiper。  
   - 目标：先把“冷启动 1~3 步区间”单独优化，而非只看终点分。

2. **把 memory 从“存内容”改为“存决策痕迹”（两周迭代）**  
   - 记忆项加入：失败查询改写历史、已证伪路径、跨文档桥接线索。  
   - 策略：若连续两步仅做近义改写且无新证据，触发检索策略切换（BM25↔dense↔视觉定位）。  
   - 目标：降低 navigation/retrieval failure 占比与长轨迹失败率。

3. **设计多目标 RL 奖励（一个月实验）**  
   - 奖励 = 答对分 + 证据对齐分（Page/Doc F1 proxy）- 过度步数惩罚 - loop 惩罚。  
   - 增加“预算条件策略”训练（N≤3、N≤5、N≤10 三档）。  
   - 目标：避免只在大预算下好看，提升低预算可用性。
