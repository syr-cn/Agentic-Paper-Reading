## 4) Experiments（实为估算框架；保留“实验数字+缺失标注”）
### 4.1 Setup（可提取）
1. **单位员工画像**：100 tok/s；160h/月；57.6M tok/月；利用率 50%。
2. **方法-1 API 收入**：年化收入约 **$90B**（OpenAI ~25B, Anthropic ~19B, others ~46B）。
3. **方法-2 GPU**：全球 **~6M H200e**；60% 用于推理；8 卡节点峰值 3000 tok/s；50% 利用率后 1500 tok/s；每节点约 15 数字员工。
4. **方法-3 电力**：推理可用电力 **~10GW**；每数字员工 **~650W**（含 PUE）。
5. **能力与效率参考**：OSWorld 成功率 agent **72.7%** vs human **72.4%**；OSWorld-Human 步数 **1.4–2.7×**；后程单步时延可达前程 **3×**。

> 缺失：
> - H200e 等效规则、吞吐测试条件、利用率统计窗口、误差条。**原文未给出可提取数字**。

### 4.2 Main results（数字汇总）
| 主题 | 数字 | 解读 |
|---|---:|---|
| 2026 全球可支撑规模（GPU约束） | **~6.8M** | 供给瓶颈先在 GPU/封装链 |
| 电力可支撑规模 | **~15.4M** | 电力短期非最先约束 |
| API 收入可支撑规模 | **5–9M** | 与 GPU 口径同量级，互相校验 |
| API-friendly 产出效率（作者判断） | **10–100× human** | 代码/文档类价值最大 |
| GUI-heavy 效率 | 成功率近人类，但步数 **1.4–2.7×** | “能做”不等于“做得快” |
| 2030 推演 | **1.4B, $72/月** | 到“全球知识工作者量级” |
| 2035 推演 | **72B, $4/月** | 强乐观、依赖多重假设同成立 |

### 4.3 Analysis（>=3）
1. **现象**：三口径（API/GPU/电力）给出同量级结论（5M–15M），GPU 最紧（6.8M）。  
   **解释**：需求、供给、能源三侧交叉后，短板在 GPU 及上游 CoWoS/HBM。  
   **我的判断**：方法价值高于具体点估；“先封装内存、后电力”的排序有现实解释力。

2. **现象**：OSWorld 成功率接近人类，但效率差明显（1.4–2.7×步数 + 后程时延恶化）。  
   **解释**：GUI 任务存在长交互链、观测噪声和步骤累积延迟。  
   **我的判断**：企业落地 KPI 应该看 **TTC（time-to-complete）/ step-efficiency**，而非只看成功率。

3. **现象**：终端价格从基础设施成本被抬升到高倍数（文中给出约 24×链路）。  
   **解释**：多层供应链毛利叠加，而非纯边际算力成本定价。  
   **我的判断**：只做 token 成本优化不够；要同步关注中上层产品与渠道毛利压缩。

4. **现象**：2030→2035 增长极陡（1.4B→72B）。  
   **解释**：作者假设“能力追赶→蒸馏释放→自我强化”三阶段连续成立。  
   **我的判断**：方向可讨论，但区间极敏感；缺乏显式置信区间与灵敏度曲线，**原文未给出可提取数字**。

## 5) Cases（>=2）
### Case 1：API-native 软件团队（偏乐观可落地）
- 场景：代码生成、测试、文档、数据分析全 API 化。
- 套用文中参数：数字员工 100 tok/s、57.6M tok/月；frontier 成本区间约 $691–$1,440/月。
- 启示：若流程本身 API-friendly，单位人类管理者可编排多 agent 并行，产出弹性显著。
- 风险：质量保障与责任归因机制若不完善，会吞掉部分效率红利。

### Case 2：GUI-heavy 企业后台（偏保守）
- 场景：ERP、表单、传统桌面软件、多窗口切换。
- 证据：成功率接近人类，但步数 1.4–2.7×，后程时延可 3×。
- 结论：在不改造系统接口前，“可自动化”不等于“更便宜更快”；先做 API 化改造比盲目堆模型更关键。

### Case 3：成本传导链视角（采购/财务）
- 场景：企业只盯电费与 GPU 折旧。
- 文中提醒：电费占比很低（约 $30/月量级），终端价受多层毛利与商业模式影响更大。
- 结论：预算优化应覆盖 SaaS 合同结构、计费模型（token→结果导向）与组织流程，而非仅底层算力。

## 6) Actionable next steps（面向我们）
1. **建立“成功率+时长+成本”三维基准**：每个任务记录 success / TTC / token / 每任务美元成本，避免单指标误导。
2. **做 API-friendly 改造优先级表**：把 GUI 链路最长、等待最多的流程先 API 化，预期收益通常高于单纯换更大模型。
3. **引入 sensitivity dashboard**：对利用率、模型单价、毛利率做 ±20% 扰动，输出容量与成本区间（补足本文缺失）。
