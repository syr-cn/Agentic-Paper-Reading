## 4 Experiments（严格说是“估算框架与情景推演”，非实验论文）
### 4.1 Experimental setup（原文可提取设置）
1. **单位员工算力画像**：100 tok/s，160h/月，57.6M tok/月，SaaS 利用率 50%。
2. **方法 1（市场约束）**：全球主要 AI 服务商年化收入约 **$90B**；按输出单价换算每名数字员工月成本：
   - $12/M tok → **~$691/月**；
   - $15/M tok → **~$864/月**；
   - $25/M tok → **~$1,440/月**。
3. **方法 2（硬件约束）**：
   - GPU 存量：~6M H200e；
   - 推理占比：60%（3.6M）；
   - 8 卡节点峰值：3000 tok/s；
   - 50% 利用率后有效吞吐：1500 tok/s；
   - 每节点支撑员工：15；
   - 总规模：~6.8M。
4. **方法 3（能源约束）**：
   - LLM 推理可用电力：10GW；
   - 每员工平均功耗（含 PUE）：650W；
   - 总规模：~15.4M。
5. **效率/能力参考设置**：
   - OSWorld：agent 成功率 **72.7%** vs human **72.4%**；
   - OSWorld-Human：agent 步数 **1.4–2.7×** 人类；后期单步时延可达前期 **3×**。
6. **供应链/成本设定**：
   - CoWoS 2026 产能：~680k wafers/year；
   - 3nm+5nm 年产能：~4.3M wafers/year；
   - 成本链路：基础设施约 **$125/月**，终端约 **$2,950/月**（约 24×）。

### 4.3 Analysis（至少 3 条：现象 + 解释 + 我的判断）
1. **现象：** 三口径估算（API 收入 / GPU / 电力）都落在同一数量级（约 5M–15M），而 GPU 给出更紧上限（~6.8M）。  
   **解释（作者）：** 市场需求、电力、硬件供给互相校验后，当前最硬瓶颈是 GPU（尤其其上游 CoWoS/HBM）。  
   **【我的判断】：** 这个交叉框架是本文最有价值的方法贡献；即便绝对值有误差，**“先看封装与内存，再看电力”**这一排序判断是可信的。

2. **现象：** OSWorld 中 agent 成功率已接近人类（72.7% vs 72.4%），但执行效率明显偏低（1.4–2.7×步数，后期时延恶化）。  
   **解释（作者）：** GUI-heavy 任务存在交互链路长、观测噪声大、步骤累积延迟等问题，所以“成功率追平”不代表“吞吐追平”。  
   **【我的判断】：** 这直接解释了为什么企业里“能自动化”的比例没有想象中高；对我们做 agent 系统，关键 KPI 应该是 **time-to-complete 与 step-efficiency**，不只是 success rate。

3. **现象：** 成本链路从基础设施到终端价格存在约 24×加价（$125 → $2,950/月），电费仅约 $30/月。  
   **解释（作者）：** 价格不是由边际算力成本单独决定，而是由供应链各层毛利叠加决定。  
   **【我的判断】：** 这点很重要：我们如果只优化模型 token 成本，可能只碰到“底层地板”；真正决定大规模 adoption 的是中上层产品与渠道结构是否压缩毛利。

4. **现象：** 2030–2035 预测极端陡峭（1.4B → 72B），同时单价降至 $72 → $4/月。  
   **解释（作者）：** 依赖三阶段机制：能力追赶（Jevons）→ 蒸馏扩散（成本释放）→ 自我强化（AI 改进芯片/电力/调度）。  
   **【我的判断】：** 方向上可能成立，但数值上高度敏感；文中未提供置信区间与敏感性曲线，**原文未给出可提取数字**（如不同利用率/毛利压缩速度下的区间结果）。

## 5 Why it matters for our work
对我们做 agent memory / long-context / multimodal RL，这篇的启发很直接：
- **Memory 不是“越长越好”**，而是要看单位产出成本（tok/s、cache 命中、有效步骤数）；
- **Long-context 的价值需绑定任务闭环时间**，否则只是堆 token；
- **Multimodal RL 的主战场在 GUI-heavy 场景提效**，要把“步数倍增 + 时延累积”这两个现实瓶颈打掉。

## 6 Actionable next step（可执行，面向 memory / long-context / multimodal RL）
1. **Agent Memory 成本-效果基准**（两周内可落地）  
   建一个 benchmark：固定任务集下记录 `成功率 / 完成时长 / 总token / cache命中率 / 单任务成本`；目标不是只看准确率，而是找“最便宜可用记忆配置”。

2. **Long-context 分层记忆调度实验**（一个迭代周期）  
   做三档策略 A/B/C：全量上下文、摘要记忆、检索记忆；比较在 2h+ 长任务中的 step 数和 wall-clock time，重点验证是否能把后程时延增长压平。

3. **Multimodal RL 的 GUI 步数压缩项目**（季度目标）  
   以 OSWorld 类任务为模板，优化目标设为：在成功率不降前提下，把 interaction steps 从当前基线降到 **≤1.2× human**；同时跟踪每步时延斜率，避免“后程 3×”恶化。
